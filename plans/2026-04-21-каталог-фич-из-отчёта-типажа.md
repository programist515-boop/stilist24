## Параллелизация Этапа 1

Ключевой принцип: **YAML-контент и код-сервис каждой фазы — это две разные задачи**. Контент с `pending_` атрибутами (как в [flamboyant_gamine.yaml](../ai-stylist-starter/config/rules/reference_looks/flamboyant_gamine.yaml)) пишется впереди кода, сервисы прототипируются на моках. Единственный реальный блокер — миграция `Item` в Фазе 0.

### Треки

| Трек | Профиль исполнителя | Что делает |
|---|---|---|
| **A. Инфра** | backend-инженер | Фаза 0 (миграция + CV) → движки Фаз 2, 1, 6, 3, 7 |
| **B. Контент** | стилист + контент-инженер | Все YAML на `pending_` атрибутах: Фазы 1, 2, 3, 4, 6, 7, 8 |
| **C. Color try-on** | CV-инженер | Фаза 5 целиком — полностью изолированная фича |
| **D. Identity DNA** | контент + простой фронт | Фаза 8 целиком — расширение YAML + UI-карточка |
| **E. Фронт** | frontend-инженер | UI под все фазы по мере готовности API |

### Три волны

**Волна 1 — старт в день 1, всё параллельно:**

- Трек A: Фаза 0 — миграция `Item`, CV-экстрактор 14 новых полей
- Трек B: YAML-наполнение сразу для **Фаз 1, 2, 3, 4, 6, 7** (все 14 категорий, силуэт, стоп-листы ткани/принты/цвет/детали, справочник стилей, 12 YAML референсных луков, цветовые схемы) — всё на `pending_` атрибутах
- Трек C: **Фаза 5** целиком (rembg+HSV, API, фронт) — не зависит ни от чего в плане
- Трек D: **Фаза 8** целиком (`associations`, `motto`, `celebrity_examples[]`, карточка «Кто ты стилистически»)

Результат Волны 1: ~70% работы Этапа 1 закрыто по контенту. Фазы 5 и 8 можно релизить в прод — они не ждут остальных.

**Волна 2 — после того как Фаза 0 в main:**

- Трек A: `category_rules_service` (Фаза 2 движок) + `silhouette_rules_service` (Фаза 1 движок) + интеграция `style_tag[]` в `outfit_engine` (Фаза 6 движок)
- Трек B параллельно: `color_combination_service` (Фаза 4 движок) — не зависит от атрибутов, может делаться параллельно

**Волна 3 — композиционные фичи, требуют данных Волны 2:**

- `stop_lists_service` + explainer (Фаза 3 движок) — собирает данные из category_rules и силуэта
- `reference_matcher` (Фаза 7 движок) — нужны атрибуты + category_rules для валидации матчей
- Финальная интеграция всего в `outfit_engine`

### Зависимости в виде графа

```
Волна 1 (параллельно):
  ┌── A: Фаза 0 (блокер)
  ├── B: YAML Фаз 1,2,3,4,6,7 (pending_ атрибуты)
  ├── C: Фаза 5 целиком ──────────────► можно релизить
  └── D: Фаза 8 целиком ──────────────► можно релизить

Волна 2 (после Фазы 0):
  ┌── A: движки Фаз 2, 1, 6
  └── B: движок Фазы 4 (параллельно)

Волна 3 (после Волны 2):
  ├── движок Фазы 3 (нужна Ф.2)
  └── движок Фазы 7 (нужны Ф.0 + Ф.2)
```

### Практические режимы работы

- **С 1 инженером:** последовательно A-трек (0 → 2 → 7 → остальное), параллельно в фоне B-контент без переключения контекста. Треки C и D — откладываются или отдаются подрядчику.
- **С 2 инженерами:** один на A (инфра → движки), второй на C (изолированный color try-on). Контент — стилист / подрядчик.
- **С 3+ инженерами:** полный параллелизм — треки A, C, D сразу + контент в Треке B.

### Быстрые победы (ship early)

Фазы, которые можно отгрузить в прод **до завершения Волны 2**:

1. **Фаза 5 (color try-on)** — замкнутая, видимый UX-wow, ~1 неделя инженера.
2. **Фаза 8 (identity DNA)** — контент + карточка, 1–2 дня, моментально повышает «комплексность» в UX.
3. **Фаза 4 YAML-часть (цветовые схемы)** — без сервиса, как справочник для будущей логики.
