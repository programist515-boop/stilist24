## Контекст

Сейчас прод стоит на одном VPS `194.67.99.214` (`/home/ubuntu/stilist24-com`):
весь стек крутится в `docker-compose.prod.yml` (api, web, db, redis, minio,
createbuckets), хостовой nginx терминирует TLS и склеивает три сервиса под
одним доменом `stilist24.com` (`/`, `/api/`, `/s3/`). CI/CD (`.github/workflows/deploy.yml`)
пушит код по SSH и пересобирает compose.

Цель — съехать на Amvera (PaaS), чтобы избавиться от ручного админства VPS
(nginx, бэкапы, мониторинг, обновления Ubuntu, ручные restore'ы при инцидентах
вроде 2026-04-25). Amvera возьмёт на себя HTTPS, рестарты, healthcheck'и,
managed-БД и логи.

**Решения, согласованные с пользователем:**
- Аккаунт Amvera уже есть, проект пустой.
- URL — субдомены: `stilist24.com` → web, `api.stilist24.com` → api,
  `media.stilist24.com` → Yandex Object Storage.
- Хранилище — Yandex Object Storage (внешний S3, не MinIO).
- Production-данные переносим целиком (Postgres + bucket с фото).

```
Yandex Cloud DNS (или текущий регистратор):
  stilist24.com           A/CNAME → Amvera web app
  www.stilist24.com       CNAME   → stilist24.com
  api.stilist24.com       CNAME   → Amvera api app
  media.stilist24.com     CNAME   → bucket.storage.yandexcloud.net

Amvera проект "stilist24":
  ┌──────────────────────────────────────────┐
  │ stilist24-web   (Docker, Next.js)        │  → stilist24.com
  │   build-arg NEXT_PUBLIC_API_URL=         │
  │     https://api.stilist24.com            │
  ├──────────────────────────────────────────┤
  │ stilist24-api   (Docker, FastAPI)        │  → api.stilist24.com
  │   alembic upgrade head на старте         │
  │   подключён к managed Postgres + Redis   │
  ├──────────────────────────────────────────┤
  │ Postgres 16    (managed service)         │  internal
  │ Redis 7        (managed service)         │  internal
  └──────────────────────────────────────────┘

Yandex Cloud:
  Object Storage bucket "ai-stylist"          → media.stilist24.com
    (public-read для GET /<key>)
  Service Account с ролью storage.editor       → ключи в S3_ACCESS_KEY/SECRET
```

`createbuckets`-сервис из compose выпиливаем — bucket создаётся в Yandex руками
один раз. MinIO больше не нужен.

## Phase 0 — Подготовка ресурсов (Amvera + Yandex)

### Amvera

1. В существующем проекте Amvera создать **managed Postgres 16** —
   сохранить connection string (хост, порт, пользователь, пароль, имя БД).
2. Создать **managed Redis 7** — сохранить connection string.
3. Подготовить (но пока не деплоить) два Docker-приложения:
   `stilist24-api` и `stilist24-web`. Привязать к GitHub-репозиторию
   `stilist24` или подготовить git-remote Amvera для push'а.

### Yandex Cloud

4. Создать **Object Storage bucket** `ai-stylist`, доступ — public-read
   на объекты (`GET <key>` без авторизации, как было в MinIO с
   `mc anonymous set download`).
5. Создать **Service Account** с ролью `storage.editor`, выпустить
   статические ключи доступа → `S3_ACCESS_KEY`, `S3_SECRET_KEY`.
6. (Опционально, для красивых URL) Привязать к bucket'у CNAME
   `media.stilist24.com` через Yandex Cloud DNS + сертификат через
   Cloud CDN или Smart Web Security. Если решим не возиться с
   сертификатом для media — оставляем `S3_PUBLIC_BASE_URL=https://storage.yandexcloud.net/ai-stylist`.

**Критерий выхода фазы:** все connection strings и ключи собраны в одном
безопасном месте (1Password / GitHub Secrets staging-prefix), bucket пуст
и доступен на чтение анонимно.

#### B. Yandex Cloud (Object Storage)

Войти в https://console.cloud.yandex.ru. Нужен аккаунт с привязанной
картой (есть бесплатный grant + минимальные тарифы).

**B.1. Создать каталог (folder)**
Если в облаке ещё нет каталога: Cloud → ваш облако → «Создать каталог»
→ имя `stilist24`. Все ресурсы создавать в нём.

**B.2. Создать service account**
1. IAM → «Сервисные аккаунты» → «Создать сервисный аккаунт».
2. Имя: `stilist24-storage-sa`.
3. Роли: **`storage.editor`** (read+write на объекты bucket'а).

**B.3. Выпустить статический ключ для S3 API**
1. Открыть карточку сервисного аккаунта `stilist24-storage-sa`.
2. «Создать новый ключ» → **«Создать статический ключ доступа»**.
3. Скопировать `access_key_id` (это `S3_ACCESS_KEY`) и `secret`
   (это `S3_SECRET_KEY`) — **secret показывается один раз**, сохранить
   в 1Password.

**B.4. Создать bucket**
1. Object Storage → «Создать bucket».
2. Имя: `ai-stylist` (если занято глобально — добавить уникальный
   суффикс типа `ai-stylist-stilist24` и потом обновить `S3_BUCKET` в
   env api).
3. Класс хранилища: **Standard**.
4. Доступ: «Ограниченный».
5. Размер: можно поставить лимит ~10 ГБ для начала (защита от
   неожиданного счёта).

**B.5. Включить public-read для GET объектов**

У Yandex Object Storage нет одной кнопки «public». Делаем bucket policy:
1. Открыть bucket `ai-stylist` → «Настройки» → «Политика доступа».
2. Вставить JSON:
   ```json
   {
     "Version": "2012-10-17",
     "Statement": [
       {
         "Sid": "PublicReadGetObject",
         "Effect": "Allow",
         "Principal": "*",
         "Action": "s3:GetObject",
         "Resource": "arn:aws:s3:::ai-stylist/*"
       }
     ]
   }
   ```
3. Сохранить.

Проверка: `curl -I https://storage.yandexcloud.net/ai-stylist/test.txt`
должен возвращать 404 (объекта нет, но bucket видит запрос без 403).

**B.6. CNAME `media.stilist24.com` — ОТЛОЖИТЬ до Phase 4**

Сначала прогоняем e2e на `https://storage.yandexcloud.net/ai-stylist/...`,
потом, если хотим красивые URL, привязываем CNAME через Cloud CDN
(требует сертификата). Можно жить и без CNAME.

### 4.1. Снизить TTL текущих A-записей в reg.ru — **первый шаг, делать сейчас**

В личном кабинете reg.ru → `Домены и услуги → stilist24.com → DNS-записи (DNS-серверы и управление зоной)`:
- A `@` (или `stilist24.com`) `194.67.99.214` → **TTL 300 сек** (5 мин)
- A `www` `194.67.99.214` → **TTL 300 сек**

Why: при cutover'е дефолтный TTL 3600+ сек кэширует VPS-IP до часа. Снижение до 5 мин = быстрый rollback.

### 4.2. Custom domains в Amvera UI (3 шт)

Аналогично Phase 2, но с **типом «Кастомное доменное имя»** (НЕ «Бесплатный домен Амвера»):

1. **`stilist24-api → Домены → Создать доменное имя`:** HTTPS, Кастомное, имя `api.stilist24.com`. Применить.
2. **`stilist24-web → Домены → Создать доменное имя`:** HTTPS, Кастомное, имя `stilist24.com` (БЕЗ www, БЕЗ https://). Применить.
3. **`stilist24-web → Домены → Создать доменное имя`:** HTTPS, Кастомное, имя `www.stilist24.com`. Применить.

После Применить Amvera в столбце **«Действие»** покажет инструкцию для DNS — это могут быть:
- `CNAME → <id>.amvera.io` (для api и www) — типичный путь
- `A → 158.160.116.999` (для apex `stilist24.com`) — CNAME на apex запрещён по RFC

**Скриншот раздела Домены обоих приложений → пользователь даёт точные значения мне.**

### 4.3. DNS-записи в reg.ru на Amvera

## Риски и open questions

- **OpenAI direct 403 с RU-IP** — Amvera в РФ, проблема не решится переездом.
  Это не блокер: `USE_CV_CATEGORY_CLASSIFIER=false`, фронт показывает ручной
  dropdown. Решение откладываем до отдельного плана (kie.ai OpenAI-endpoint /
  OpenRouter / EU-relay).
- **Размер api-образа** — Dockerfile.prod ставит OpenCV/MediaPipe + скачивает
  модели в `download_models.py`. Образ ~1.5 ГБ, билд 8–12 минут. Нужно
  убедиться, что у тарифа Amvera хватает таймаута на билд и места под
  layer'ы.
- **CNAME на apex** `stilist24.com` — некоторые регистраторы не поддерживают
  CNAME на apex (только A). Проверить заранее, у Amvera должен быть
  IP-таргет как fallback.
- **Финальная дельта в Phase 4.3** — между последним `pg_dump` и DNS-cutover'ом
  пользователи могут что-то записать на VPS. Варианты: (а) поставить VPS в
  read-only режим за 5–10 минут до cutover'а, (б) принять микро-потерю
  данных за окно cutover'а (закрытая бета — допустимо), (в) сделать
  логическую репликацию VPS→Amvera (overkill для бета-версии). Рекомендую (б)
  с уведомлением пользователей за 24 часа.
- **Yandex Object Storage path-style** — у нас в config.py `s3_force_path_style: True`.
  Yandex Storage поддерживает обе схемы, path-style работает. Если в коде
  где-то жёстко используется virtual-hosted style — поправим точечно
  (но в boto3 это конфигурируется флагом, в коде пробежался — таких мест нет).

## Архитектурное решение: per-object canned ACL вместо bucket policy

**Принято 2026-04-29. Отличается от того, что было в исходном плане
(B.5 «применить bucket policy PublicReadGetObject»).**

### Контекст

Yandex Object Storage не даёт роли `storage.editor` право менять bucket
policy и bucket-level ACL — это требует `storage.admin`. Попытка
`PUT /<bucket>?policy` и `PUT /<bucket>?acl` с публичной политикой
возвращает HTTP 403 Access Denied.

### Развилка

Три варианта обеспечить публичный read объектов:

1. **Эскалировать роль SA до `storage.admin`** — лишние права, которые
   проекту в runtime не нужны (мы не меняем bucket settings из приложения,
   только uploads/deletes). Безопаснее держать минимум.
2. **Toggle bucket setting «Чтение объектов: Для всех» через Yandex UI** —
   разовый ручной клик, не автоматизируется, легко забыть при пересоздании
   bucket'а или миграции на другой регион.
3. **Per-object canned ACL `public-read`** — каждый PUT object несёт
   header `x-amz-acl: public-read`. Объект становится publicly readable
   индивидуально. Право `s3:PutObjectAcl` входит в `storage.editor`,
   так что эскалация не нужна.

### Решение

Выбрали (3). Реализация:

1. В [`config.py`](../ai-stylist-starter/app/core/config.py) добавлено
   поле:
   ```python
   s3_object_acl: str | None = None
   ```
   Дефолт `None` — поведение для MinIO/AWS не меняется.

2. В [`storage.py`](../ai-stylist-starter/app/core/storage.py)
   `S3StorageBackend.put()` собирает kwargs и условно добавляет
   `ACL=settings.s3_object_acl`:
   ```python
   kwargs = {"Bucket": ..., "Key": key, "Body": data, "ContentType": content_type}
   if settings.s3_object_acl:
       kwargs["ACL"] = settings.s3_object_acl
   self._get_client().put_object(**kwargs)
   ```
   `_DefaultSettings` тоже получил поле, чтобы импорт `app.core.storage`
   без pydantic-settings (тестовое окружение) продолжал работать.

3. В env прода:
   ```
   S3_OBJECT_ACL=public-read
   ```

### Совместимость

- **MinIO (dev):** в `.env`/`.env.example` `S3_OBJECT_ACL` не задан →
  `kwargs["ACL"]` не передаётся → поведение как было (bucket-level
  public-read настраивается через `mc anonymous set download`,
  `createbuckets` сервис в docker-compose).
- **AWS S3:** канонический `public-read` ACL поддерживается, но требует
  отключённого Block Public Access. Если в будущем переедем на AWS — либо
  выключаем BPA на bucket'е, либо переключаем на presigned URLs
  (механизм уже есть в `S3StorageBackend.public_url()` — fallback на
  presign, если `s3_public_base_url` не задан).
- **Yandex (prod):** `S3_OBJECT_ACL=public-read` обязательно, иначе фото
  пользователя не откроются в браузере (403).
